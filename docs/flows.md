# Scope Shield — 核心流程与状态机

## 范围

MVP 的所有用户旅程、状态机、失败路径和恢复行为。

## 非范围

- 团队协作流程
- 云同步流程
- 第三方集成流程

## 假设

- 单用户操作，无并发冲突
- 所有操作在客户端完成

## 依赖

- docs/mvp-prd.md 功能需求
- docs/data-model.md 实体定义

---

## 最短成功流

```
打开 App
  → 首次：看到 Demo 项目 → 浏览体验
  → 点击"新建项目" → 填名称+日期 → 创建
  → 点"+ 添加需求" × 3 → 各填名称+天数
  → [日常使用] 点"记录变更" → 选类型 → 填信息 → 保存
  → 图表自动更新
  → 点"导出图片" → 选尺寸 → 下载 PNG
  → 发到工作群
完成 ✓
```

**预计耗时**：首次创建 3 分钟，日常记录变更 30 秒，导出 5 秒。

## 核心流程

### Flow-1: 创建项目 + 录入需求

```
[首页/侧边栏]
  │
  ├─ 点击"新建项目"
  │    → 弹出表单：项目名称(必填) + 开始日期(默认今天)
  │    → 验证：名称不能为空
  │    → 保存 → 跳转项目详情页
  │
  └─ [项目详情页]
       │
       ├─ 点击"+ 添加需求"
       │    → 内联表单：名称(必填) + 天数(必填, ≥1)
       │    → 可选：设置依赖（下拉选已有需求）
       │    → 保存 → 需求出现在列表
       │    → 总工期实时重算
       │
       ├─ 重复添加...
       │
       └─ 完成 → 图表开始渲染
```

**失败路径**：
- 名称为空 → 输入框红色高亮 + 提示文字
- 天数非法（< 0.5 或非 0.5 步进）→ 输入框红色高亮
- 循环依赖 → Toast 提示"不能设置循环依赖（A→B→...→A）"，依赖选择不保存

**恢复**：修正输入后可重新提交

### Flow-2: 记录变更（核心流，≤30秒）

```
[项目详情页]
  │
  ├─ 点击"记录变更"按钮
  │    → 弹出变更表单
  │
  ├─ Step 1: 选类型
  │    → [加天数] [新增需求] [砍需求] [需求补充] [调优先级] [暂停] [恢复]
  │
  ├─ Step 2: 填信息（根据类型动态显示字段）
  │    ┌─ 加天数：选需求(仅active) + 增加天数(≥0.5)
  │    ├─ 新增需求：名称 + 天数(≥0.5)
  │    ├─ 砍需求：选需求(仅active)
  │    ├─ 需求补充：选需求(所有状态) + 子类型(3选1) + 天数(≥0，允许0) + 描述(必填)
  │    ├─ 调优先级：选择需求 + 选择新位置（1-based 序号）
  │    ├─ 暂停：选需求(仅active) + 剩余天数(数字，默认=currentDays，≥0.5)
  │    └─ 恢复：选已暂停的需求
  │
  ├─ Step 3: 责任人
  │    → 角色下拉（产品经理默认 / 领导 / 测试 / 其他）
  │    → 人名输入（可选，有历史建议）
  │
  ├─ Step 4: 描述（必填，一句话）
  │
  └─ 保存
       → 变更记录入库
       → 快照自动保存
       → 需求状态更新
       → 总工期重算
       → 图表实时刷新
       → 表单关闭
```

**失败路径**：
- 描述为空 → 阻止保存，高亮描述字段
- 天数非法 → 阻止保存，高亮天数字段
- 暂停已暂停的需求 → 按钮置灰不可选
- 恢复非暂停的需求 → 不在列表中出现

### Flow-3: 导出图片

```
[项目详情页 — 图表区域]
  │
  ├─ 点击"导出图片"
  │    → 弹出尺寸选择：手机 390px(默认) / 桌面 800px / 自定义
  │    → 点击"导出"
  │
  ├─ 系统渲染导出版图表（Apple 亮色风格）
  │    → 渲染完成 → 自动下载 PNG
  │
  └─ 完成
```

**失败路径**：
- 无需求且无变更记录 → 导出按钮置灰；无需求但有变更记录（需求全部硬删除）→ 仍可导出（显示变更段标注"已删除"）
- 渲染失败（极罕见）→ 提示"导出失败，请重试"
- 浏览器不支持 Canvas → 提示"请使用 Chrome/Safari/Firefox"

### Flow-4: JSON 备份与恢复

```
[设置页面]
  │
  ├─ 导出数据
  │    → 序列化所有数据为 JSON
  │    → 自动下载文件：scope-shield-backup-{YYYY-MM-DD}.json
  │
  └─ 导入数据
       → 选择文件
       → 验证 JSON 格式和结构
       ├─ 有效 → 弹窗确认"将覆盖当前数据"
       │          → 确认 → 清空 + 写入 → 刷新页面
       │          → 取消 → 什么都不做
       └─ 无效 → 提示"文件格式不正确" → 无操作
```

### Flow-5: 编辑变更记录

```
[变更记录区域]
  │
  ├─ 悬停变更行 → 点击"编辑"按钮
  │    → 打开编辑变更弹窗（预填已有数据）
  │
  ├─ 可编辑字段（所有类型通用）：
  │    → 角色、人名、描述、日期、天数
  │
  ├─ 类型更改规则：
  │    ├─ new_requirement 类型：type 和 targetRequirementId 不可更改（置灰）
  │    └─ 其他类型：可更改 type（但不可改为 new_requirement）和 targetRequirementId
  │
  ├─ 动态字段联动（type 更改时）：
  │    ├─ 改为 cancel_requirement → daysDelta 字段隐藏/置灰（系统自动计算为 -currentDays，与新建流程一致）
  │    ├─ 改为 pause → 显示"剩余天数"字段（默认=目标需求 currentDays），daysDelta 隐藏
  │    ├─ 改为 resume → daysDelta 隐藏（系统自动设为 0）
  │    ├─ 改为 reprioritize → 显示"新位置"下拉，daysDelta 隐藏
  │    ├─ 改为 add_days → 显示 daysDelta 输入框（用户填写，≥0.5）
  │    └─ 改为 supplement → 显示补充子类型(3选1) + daysDelta 输入框(≥0，允许0) + 描述(必填)
  │
  └─ 保存
       ├─ 仅修改描述/角色/人名 → 直接更新记录，不触发 replay
       └─ 修改 type/targetRequirementId/daysDelta/date/metadata中影响计算的字段（pause.remainingDays, reprioritize.fromPosition/toPosition）→ 触发 snapshot replay 重算
            → 删除所有旧 snapshot → 重新应用所有变更 → 生成新 snapshot
            → 需求状态和总工期更新
            → 图表实时刷新
```

**失败路径**：
- 描述为空 → 阻止保存
- 天数非法 → 阻止保存
- 目标需求已不存在（悬空引用）→ replay 时跳过该条

### Flow-6: 编辑需求

```
[需求列表区域]
  │
  ├─ 悬停需求行 → 点击"编辑"按钮
  │    → 需求行就地展开为内联编辑表单
  │
  ├─ 可编辑字段：名称、天数（originalDays）、依赖
  │
  └─ 保存
       ├─ 仅修改名称 → 直接更新，不触发任何重算
       ├─ 修改依赖 → 触发调度器重算（非 replay），更新总工期
       └─ 修改 originalDays →
            ├─ 该需求无变更记录 → currentDays = originalDays 直接同步，调度器重算
            └─ 该需求有变更记录 → 触发完整 snapshot replay
                 → 基线重置（使用新 originalDays）
                 → 按序重新应用所有变更
                 → currentDays 基于新 originalDays + 变更累加
                 → 总工期重算
```

**失败路径**：
- 名称为空 → 阻止保存
- 天数 < 0.5 → 阻止保存
- 循环依赖 → Toast 提示，依赖不保存

### Flow-7: Demo 项目恢复（重置）

```
[已归档分组]
  │
  ├─ 用户点击 Demo 项目的"恢复"按钮
  │
  ├─ 系统检查 project.isDemo === true
  │    ├─ true → 执行 Demo 重置流程：
  │    │    1. 删除该项目下所有需求、变更、快照
  │    │    2. 重写为初始 Demo 数据（architecture.md §Demo 数据）
  │    │    3. project.status = "active"
  │    │    4. 跳转到 Demo 项目详情页
  │    └─ false → 普通恢复：project.status = "active"
  │
  └─ 侧边栏更新：项目回到"进行中"分组
```

## 状态机

### Project 状态

```
         创建
          │
          ▼
     ┌─────────┐
     │  active  │ ←──── 恢复
     └────┬─────┘
          │ 归档
          ▼
     ┌──────────┐
     │ archived │
     └──────────┘
```

| 状态 | 可执行操作 |
|------|----------|
| active | 添加需求、记录变更、导出图片、编辑、归档 |
| archived | 只读查看、恢复、导出图片 |

**无效转换**：archived → 编辑/添加/记录变更（按钮隐藏）

**MVP 范围外**：项目硬删除（永久删除项目及所有关联数据）不在 MVP 范围内。用户只能归档项目，不能删除。

### Requirement 状态

```
         添加
          │
          ▼
     ┌─────────┐    暂停     ┌──────────┐
     │  active  │ ─────────→ │  paused  │
     └──┬───┬───┘ ←───────── └──────────┘
        │   │      恢复
        │   │
   砍需求   完成(V2)
        │
        ▼
    ┌───────────┐
    │ cancelled │
    └───────────┘
```

| 状态 | 显示 | 计入工期 |
|------|------|---------|
| active | 正常 | 是 |
| paused | 暂停图标 + 灰色 | 否（冻结剩余天数，暂停时用户输入剩余天数，默认=currentDays） |
| cancelled | 灰色删除线 | 否（显示绿色节省） |

**无效转换**：
- cancelled → active/paused（已砍不可恢复，需新建）
- paused → cancelled（需先恢复再砍。**replay 状态守卫**：cancel_requirement 变更在 replay 时检查 status === "active"，非 active 则跳过）

### Change 状态

变更记录无状态流转，创建后可编辑或删除。删除为硬删除（确认后不可恢复）。

## 自动计算逻辑

### 总工期计算（BR-4）

```
1. 收集所有 active 状态的需求（paused/cancelled 需求不参与）
2. 构建依赖图。依赖松弛规则：若依赖目标为 paused 需求，调度时**运行时**视为无依赖（startDay=0），但 dependsOn 字段**不修改**（恢复后依赖自动恢复）；若依赖目标为 cancelled 或已硬删除需求，dependsOn **已被永久清除为 null**（见§需求删除重算和 data-model.md §依赖松弛规则），调度时自然无依赖
3. 拓扑排序
4. 计算每个需求的最早开始时间和最晚结束时间
5. 总工期 = max(所有需求的 endDay)（即最长路径的终点）
```

### 依赖顺延（BR-5）

```
当需求 A 的工期变化 delta 天时：
→ 实际实现：调度器以拓扑排序 + 最长路径算法重算所有需求的 startDay/endDay（见 architecture.md §scheduler.ts），自动体现顺延效果

概念模型（简化描述）：
1. 找到所有直接依赖 A 的需求 B1, B2, ...
2. 对每个 Bi：开始日期随 A 的 endDay 变化而顺延
3. 递归：Bi 的依赖者也顺延
4. 生成提示消息列表
```

### 优先级重排（BR-6）

```
当需求 X 的优先级调整到位置 P 时：
1. 从当前位置移除 X
2. 插入到位置 P
3. 更新所有需求的 sortOrder（纯展示排序，不影响工期计算）
4. 记录变更："X 优先级调整到第 P 位"
```

**注意**：`sortOrder` 仅影响 UI 列表和甘特图的需求行顺序，不影响调度器的最长路径计算。调度器仅依据 `dependsOn` 计算工期。

### 变更删除/编辑重算

```
当删除或编辑变更记录时：
1. 回退所有需求到初始状态（originalDays + active）
2. 按 date + createdAt 排序剩余变更
3. 依次重新应用每条变更
4. 计算最终需求状态和总工期
5. 保存新快照
```

**触发条件汇总**：
| 操作 | 触发 replay | 触发调度器重算 |
|------|------------|--------------|
| 删除变更记录 | ✅ | — (replay 含调度) |
| 编辑变更的 type/target/daysDelta/date/metadata(影响计算的字段) | ✅ | — |
| 编辑变更的 描述/角色/人名 | ❌ | ❌ |
| 编辑需求 originalDays（有变更记录时） | ✅ | — |
| 编辑需求 originalDays（无变更记录时） | ❌ | ✅ (直接同步) |
| 编辑需求 dependsOn | ❌ | ✅ |
| 编辑需求 名称 | ❌ | ❌ |
| 删除需求（硬删除） | ❌ | ✅ |

详见 data-model.md §变更重算策略

### 需求删除重算

```
当删除需求时（硬删除）：
1. 删除需求记录
2. 相关变更的 targetRequirementId 变为悬空引用
3. 系统回写 Change.metadata.deletedRequirementName（原需求名）
4. 其他需求中 dependsOn 指向该需求的，自动清除为 null
5. 调用调度器重算总工期（不触发完整 replay，因为变更记录未改变）
```

### 路由守卫

```
当访问 /project/:id 时：
1. 查询 ID 对应项目是否存在
2. 存在 → 正常加载
3. 不存在 → 重定向到最近访问的有效项目
4. 无任何项目 → 重定向到 / → 触发 Demo 项目创建
```
