# Scope Shield — MVP PRD

## 范围

开发者工期保护工具 MVP：本地 Web 应用，记录需求变更及其工期影响，生成两种可视化图表（膨胀条 / 时间线+甘特），导出 PNG 图片。

## 非范围

- 团队协作 / 多人共享
- 云同步 / 账号系统
- Jira / 飞书 / 禅道集成
- AI 工时估算
- 移动端适配
- 复杂依赖图（MS Project 级别）
- 实时通知 / 邮件提醒

## 假设

- 用户愿意手动录入变更，30 秒录入门槛足够低
- 本地存储（IndexedDB）足够 MVP 场景
- 单用户使用，无需登录和权限管理
- 单 Tab 操作，不支持多 Tab 同时编辑同一项目（多 Tab 打开同一项目时不保证数据一致性，切换 Tab 后需手动刷新）
- 单项目最大需求数：50；单项目最大变更记录数：200。超出时提示"已达上限"并阻止添加
- 目标浏览器：Chrome 90+、Safari 16+、Firefox 100+、Edge 90+
- UI 语言：中文（MVP 不支持国际化，V2 考虑）

## 依赖

- 无外部后端服务
- 浏览器 IndexedDB API
- Canvas/SVG → PNG 导出能力

---

## 产品一句话

给 **5-15 人团队开发者**，在 **项目中途被加/改需求** 时，**用 30 秒记录变更并一键生成工期影响图**，避免延期背锅。

## 问题陈述

开发者在项目开发过程中频繁遭遇需求变更（产品经理改范围、领导加功能、测试反馈导致返工），项目延期后开发者成为第一背锅人。现有工具（Jira、飞书项目）是管理者视角，不提供"变更 → 工期影响 → 责任归属"的直观可视化。开发者缺乏一个简单的武器来自证清白。

## MVP 目标

让开发者能在 3 分钟内创建项目并录入需求，日常 30 秒记录一次变更，一键导出一张图发到工作群，让所有人 3 秒看懂"延期是谁的变更导致的"。

## 主用户

**5-15 人团队的前端/后端开发者**：有飞书但用得模糊，没有专门的甩锅手段，嘴笨不善于当面解释延期原因。使用频率：每周数次。

## 术语表

| 术语 | 定义 |
|------|------|
| **Project（项目）** | 一个有明确开始日期和原始工期的开发项目，包含需求列表和变更记录 |
| **Requirement（需求）** | 项目中的一项工作单元，有名称、预估天数、状态、可选依赖 |
| **Change（变更记录）** | 一次需求变更事件，记录类型、责任人、描述、工期影响 |
| **Snapshot（快照）** | 每次变更自动保存的项目状态副本 |
| **膨胀率** | (当前工期 - 原始工期) / 原始工期 × 100% |
| **最长路径** | 考虑并行和依赖后，决定项目总工期的关键路径 |
| **角色** | 变更责任人的分类：产品经理（默认）、领导、测试、其他 |
| **简洁版** | 图表 A — 膨胀对比条，用于日常发群 |
| **详细版** | 图表 C — 时间线故事 + 甘特条，用于多需求复杂场景 |

## 功能需求

### 项目管理

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-1 | 创建项目 | 用户输入项目名称和开始日期（默认今天），系统创建空项目并跳转到项目页 |
| FR-2 | 项目列表 | 左侧边栏显示所有项目，分"进行中"和"已归档"两组，点击切换，当前项目高亮 |
| FR-3 | 归档项目 | 用户可手动归档项目，归档后移到"已归档"分组 |
| FR-4 | 恢复项目 | 已归档项目可手动恢复到"进行中" |
| FR-5 | Demo 项目 | 首次打开时预填一个完整 Demo 项目（含需求、变更、图表），展示完整功能 |
| FR-6 | 编辑项目 | 可修改项目名称（点击进入编辑态）。开始日期 MVP 不可修改 |

### 需求管理

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-10 | 添加需求 | 点击"+ 添加需求"，填写名称和预估天数，保存后立即出现在列表中 |
| FR-11 | 编辑需求 | 可修改需求名称、天数和依赖。**天数修改**：无变更记录时 currentDays = originalDays 直接同步；有变更记录时触发 replay 重算。**依赖修改**：触发调度器重算（非 replay），更新总工期和日程 |
| FR-12 | 删除需求 | 删除需求需确认弹窗，删除后相关变更记录保留（UI 显示"需求已删除"带原名称缓存于 Change.metadata.deletedRequirementName），触发工期重算 |
| FR-13 | 简单依赖 | 可设置 A→B 单一依赖（一个需求最多依赖一个其他需求），A 延期时 B 自动顺延，系统显示提示"因 A 延期，B 被顺延 X 天" |
| FR-14 | 需求排序 | 需求列表支持拖拽排序（直接更新 sortOrder，不创建变更记录）。另可通过变更弹窗"调优先级"类型调整顺序（创建变更记录）。两种方式均不影响工期计算 |

### 变更记录

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-20 | 记录变更 — 加天数 | 选择已有需求，填写角色（默认产品经理）、可选人名、描述（必填）、增加天数。保存 ≤ 30 秒 |
| FR-21 | 记录变更 — 新增需求 | 填写新需求名称、天数、角色、人名、描述。系统自动添加需求并记录变更 |
| FR-22 | 记录变更 — 砍需求 | 选择需求标记为取消，填写角色、描述。需求显示灰色删除线，工期回收显示绿色"节省 X 天" |
| FR-23 | 记录变更 — 优先级调整 | 选择需求调整显示顺序，系统更新 sortOrder（纯展示，不影响工期），记录"X 优先级调整到第 N 位" |
| FR-24 | 记录变更 — 暂停 | 选择需求标记暂停，填写剩余天数（默认=当前天数），暂停期间不计入工期。记录暂停原因 |
| FR-25 | 记录变更 — 恢复 | 恢复已暂停需求，currentDays 替换为冻结的剩余天数，继续计入工期 |
| FR-26 | 编辑变更 | 已保存的变更记录可编辑所有字段（描述、角色、人名、日期、天数、目标需求）。**类型更改限制**：(1) `new_requirement` 类型的 type 和 targetRequirementId 不可更改；(2) 非 `new_requirement` 类型可更改 type 和目标需求，但**不可更改为 `new_requirement`**（因新增需求需关联创建实体，无法通过编辑补建）。修改 type/target/daysDelta/date/metadata 中影响计算的字段（pause.remainingDays, reprioritize.fromPosition/toPosition）时触发 snapshot replay 重算 |
| FR-27 | 删除变更 | 删除变更记录需确认弹窗，删除后触发 snapshot replay 重算。删除 `new_requirement` 变更时同时删除关联需求 |

### 图表与导出

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-30 | 简洁版图表 | 膨胀对比条：上方原始计划条，下方实际工期条（分段着色标注每次变更），底部总结延期天数和责任归属 |
| FR-31 | 详细版图表 | 时间线故事（每条变更按时间排列）+ 甘特条（每个需求的工期对比），底部汇总 |
| FR-32 | 图表切换 | 顶部 Tab 切换"简洁版 | 详细版"，实时渲染 |
| FR-33 | 导出 PNG | 一键导出当前图表为 PNG 图片，默认 390px 宽（手机），高度自适应内容 |
| FR-34 | 可选导出尺寸 | 导出时可选择手机（390px）/ 桌面（800px）/ 自定义宽度（200–2000px） |
| FR-35 | 导出风格 | 导出图片使用 Apple/iOS 亮色风格（圆角、毛玻璃、鲜艳系统色），与 App 界面风格独立 |
| FR-36 | 图表实时更新 | 任何变更保存后，图表立即重新渲染，无需手动刷新 |

### 数据管理

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-40 | 本地存储 | 所有数据存储在浏览器 IndexedDB，无需网络连接 |
| FR-41 | JSON 导出 | 一键导出所有项目数据为 JSON 文件 |
| FR-42 | JSON 导入 | 导入 JSON 文件恢复数据，导入前提示"将覆盖当前数据"确认 |

## 业务规则

| ID | 规则 | 说明 |
|----|------|------|
| BR-1 | 30 秒录入上限 | 记录一次变更的交互步骤 ≤ 4 步（选类型 → 选需求/填信息 → 填描述 → 保存） |
| BR-2 | 角色枚举 | 固定 4 类：产品经理（默认）、领导、测试、其他 |
| BR-3 | 人名记忆 | 输入过的人名自动保存，下次显示为下拉建议 |
| BR-4 | 总工期计算 | 总工期 = 考虑并行和依赖的最长路径天数 |
| BR-5 | 依赖顺延 | A→B 依赖中 A 延期 N 天 → B 开始日期自动后移 N 天，图表标注顺延提示 |
| BR-6 | 优先级重排 | 调整优先级仅改变 UI 显示顺序（sortOrder），不影响工期计算。工期由依赖关系决定 |
| BR-7 | 暂停冻结 | 需求暂停时用户输入剩余天数（默认=currentDays），冻结到 pausedRemainingDays。恢复后 currentDays=pausedRemainingDays |
| BR-8 | 砍需求回收 | 砍掉的需求标记为 cancelled，总工期按最长路径重算（砍关键路径需求才会实际缩短工期；砍非关键路径需求工期不变）。图表中显示绿色"节省 X 天"段（X = 被砍需求的 currentDays） |
| BR-9 | 快照自动保存 | 每次变更保存时，系统自动保存当前项目状态快照 |
| BR-10 | 导出人名 | 导出图片直接显示人名（如已填写） |
| BR-11 | 膨胀率公式 | 膨胀率 = (当前总工期 - 原始总工期) / 原始总工期 × 100%，四舍五入到整数。**原始总工期**：所有现存 isAddedByChange=false 需求（含 paused/cancelled 状态的需求），以其 originalDays 计算最长路径。不受变更操作（暂停/取消/加天数）影响，但会因基线需求集变化（添加新需求/硬删除需求）而重算。**当前总工期**含所有 active 需求的 currentDays |
| BR-12 | Demo 项目不可删除 | Demo 项目可归档但不可永久删除。恢复时数据重置为初始 Demo 数据（用户对 Demo 的修改将丢失） |
| BR-13 | 核心论点：0天来自开发 | 系统只记录外部变更（PM/领导/测试/其他），不提供"开发延期"类型。图表底部始终显示"0 天来自开发"，强化工具的核心价值主张 |
| BR-14 | 膨胀率除零保护 | 当 originalTotalDays = 0 时（项目无原始需求，仅由变更新增需求构成），膨胀率显示为"—"（非数字），避免除零错误。注意：原始需求被取消/暂停**不影响** originalTotalDays（冻结基线） |
| BR-15 | PNG 文件名规范 | 导出文件名格式：`scope-shield-{项目名}-{YYYY-MM-DD}.png`，项目名中特殊字符替换为下划线 |
| BR-16 | 无效路由降级 | 访问不存在的项目 ID 时，重定向到最近访问的有效项目或 Demo 项目 |
| BR-17 | JSON 备份文件名 | 导出 JSON 备份文件名格式：`scope-shield-backup-{YYYY-MM-DD}.json` |
| BR-18 | 项目不可硬删除 | MVP 不支持项目永久删除。用户只能归档项目（status → "archived"），数据保留。V2 考虑硬删除功能 |
| BR-19 | 需求删除不触发 replay | 硬删除需求后，相关变更记录保留（targetRequirementId 变为悬空引用，系统回写 metadata.deletedRequirementName）。仅触发调度器重算总工期，不触发完整 snapshot replay |
| BR-20 | 砍需求保留 currentDays | cancel_requirement 变更将需求 status 设为 cancelled，但 currentDays 保留取消前的值（不置零）。daysDelta = -currentDays（负数，用于图表绿色"节省"段宽度） |

## 成功指标

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| 首次完整流程完成率 | ≥ 70% | 从创建项目到导出图片，本地埋点 |
| 变更录入时间 | ≤ 30 秒 | 从点击"记录变更"到保存的时间差 |
| 周活跃用户留存 | ≥ 40%（7日） | 本地统计，匿名 |
| 导出图片次数 | 平均 ≥ 2次/周/活跃用户 | 本地埋点 |

**本地埋点说明**：MVP 阶段埋点为**可选实现（P2 优先级）**。若实现，仅记录到 localStorage（不新增 IndexedDB store），不上报远端。V2 考虑接入匿名分析服务。

**埋点实现路径**（若实现）：
- 存储：`localStorage` key = `scope-shield-analytics`，JSON 数组，每条含 `{ event, timestamp, data? }`
- 事件：`project_create`、`change_record`（含 `duration_ms`）、`export_image`、`json_import`、`json_export`
- 上限：最多保留最近 500 条，超出时 FIFO 淘汰
- 代码位置：`src/utils/analytics.ts`（独立模块，不影响核心功能）
- **不阻塞 MVP 上线**：即使未实现埋点，所有功能仍完整可用
