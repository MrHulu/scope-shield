## 真痛点（按优先级）

### W1 · 加需求 → 工期数字毫无变化（高频痛点）

- **现象**：demo 项目本来 23.5 天，加一条 5 天需求 → 当前工期还是 23.5、原始还是 18、膨胀率还是 +31%
- **复现**：S3 的 stats vs 加完-stats 完全一致
- **根因**（已知设计）：直接「添加需求」算入 baseline + current 双边，恰好新需求 ≤ 已有最长链 → max() 无变化
- **用户感受**：每次加需求都怀疑"是不是没保存"——这是**重复出现的 Boss 反馈**（手动复盘）
- **可选方向**：
  - (a) 加微动画 / toast 显示"已添加，不影响关键路径"
  - (b) 列表新行高亮 1.5s 提示已生效
  - (c) 改写卡片：在"当前工期"卡片旁加 +/− delta 角标，即使 0 也显式 0
- **不要**改成"直接添加 = 当作 change" — 那破坏 Scope Shield 核心模型

### W2 · 飞书未登录场景下解析慢（3.6s 无明确进度）

- **现象**：粘 URL → 点解析 → 按钮文案变"解析中" → **3.6 秒才落定**
- **真根因**：trans_simple_name 探活 + 主接口降级双 fetch 串行，未登录场景全走超时
- **影响**：用户怀疑"卡死了"
- **方向**：
  - (a) 解析按钮立即显示 spinner + "校验飞书登录…" 替代"解析中"
  - (b) 提前在 form 打开时就探活一次（背景）→ 点解析时如果已知未登录直接 url-only
  - (c) 加 ≤1s 超时强制降级到 url-only

### W3 · 「调优先级」对**原本并行**需求无影响（容易误以为坏了）

- **现象**：S5 选"目标=A、新前置=无前置"，A 本来就并行 → stats 没变
- **不是 bug**：dependsOn null → null 等价无操作
- **方向**：保存时**前置校验**——如果 newDep 等于当前 dependsOn，给 toast「前置依赖未变化，未生效」让用户立刻明白

### W4 · 截图证据 label 噪音

- 看到的是：`截图证据 (0/3，可粘贴)`
- 中英括号 + 长文案 → 改成 `截图 0/3 · 可粘贴`

### W5 · ChangeModal 密度高

- 单 modal 内有：变更类型 7 按钮 + 需求/新前置 2 select + 责任角色 + 责任人 + 日期 + 描述 + 截图 = **8 段**
- 视觉拥挤、保存按钮被压到很深
- 方向：分两步（类型 + 详情）；或分组用浅色 divider；或类型选完才出现下半部分

## ✅ 已正确实现（不要改坏）

- 首屏即用 demo 项目（无 onboarding 卡 — 商业级前要加，但 dev 期不要去掉自动跳）
- 飞书 URL **未登录**降级：占位名 + 一键登录按钮 + 仍可手动填名字保存 — Boss 多次确认这条是对的
- 调优先级新语义（需求 + 新前置）— labels 干净对齐 Boss 反馈
- 备份恢复对话框 — 弹得对，按钮文案对
- 0 console error 跑完全程（**重要！**很多前端项目首屏就有 React warning）
